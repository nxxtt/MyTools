1. Hardcoded /tmp/ paths — test_reconall.py:252, test_subdomainenum.py:116,293-327 usam /tmp/ (Unix only). Quebram no Windows. Usar tmp_path fixture do pytest.

2. Testes fazem chamadas de rede reais — test_certcheck.py:290-299 e test_graphqlattack.py:275-284 iteram dispatchers que abrem conexões SSL/TLS reais para example.com. Isso é flaky — falha sem internet, DNS ou timeout.

3. Sistema de markers do pytest subutilizado — só 1 marker (real_sleep) registrado. Faltam slow, integration, network, smoke para execução seletiva.

4. Cobertura de código (Codecov) — 4787 testes é excelente, mas sem badge de cobertura não se sabe o que está descoberto. Integrar Codecov e Coverage Gutters.

5. # type: ignore — concentrados em utils.py (6), cred.py (2), attackaudit.py (2). Cada um é um bug potencial disfarçado.

6. Rate limiter automático via Retry-After — o RateLimiter atual trata 429 com backoff, mas não parseia o header Retry-After para respeitar o tempo exato do servidor.

7. Revisão do cred.py — o gerenciador de credenciais usa keyring, mas muitos módulos aceitam senhas via --auth (argumento CLI), o que expõe no history do shell. Sugiro priorizar keyring e mascarar input.

8. attackanalysis.py usa print() puro — sem logging, sem async, 272 linhas síncronas. Para uma ferramenta que processa findings, deveria usar logging estruturado.

9. Modo --json global — hoje cada módulo formata saída de forma diferente. Um formatador JSON padronizado permitiria piping para outras ferramentas (jq, grep) de forma consistente.

10. print() sobrecarrega logging 4:1 — em módulos como corsmisconfig.py, clickjacking.py, etc., as saídas de resultado (vulnerabilidades, alertas) vão tudo via print(). Isso impede controle de verbosidade, redirecionamento, e formato estruturado. Centralizar tudo em logger.info()/logger.warning() padronizaria a saída.

11. respx instalado mas subutilizado — apenas 3 de 68 arquivos de teste web usam respx para mock HTTP. Os outros 65 usam unittest.mock diretamente, o que significa que headers, query params, e edge cases HTTP nunca são validados.

12. Cache de respostas HTTP — módulos como webrecon e attackaudit fazem múltiplas requisições ao mesmo alvo. Um AsyncClient compartilhado com cache em memória (ex: aiocache ou cachetools) reduziria drasticamente o tempo de execução.

13. Razão test:source crítica em 3 módulos:
    - http2abuse.py (1648 linhas) → apenas 200 linhas de teste (12%) — sem testes para os 7 category dispatchers
    - fileupload.py (918 linhas) → 122 linhas (13%) — sem testes para polyglot, SVG XXE, ZIP Slip
    - businesslogic.py (928 linhas) → 139 linhas (15%) — sem testes para integer overflow, race condition

14. Dockerfile multi-stage — ferramenta de pentest em container elimina "funciona na minha máquina" e facilita execução em CI/CD.

15. Boilerplate copiado em 60+ módulos — todos compartilham o mesmo esqueleto: build_parser() idêntico, print_results() idêntico, safe_asyncio_run(run_scan(...)). Uma classe base BaseScanner com método run_scan() abstrato cortaria 80-100 linhas por módulo.

16. (Muito para frente) Web UI minimalista — opção --web no main.py que sobe um servidor HTTP local (ex: FastAPI/Starlette) com formulários para cada módulo. Útil para times de segurança não-técnicos.
